# v2.7 LLM Adapter

## v2.7-A LLM Adapter

v2.7-A adds a backend LLM Adapter abstraction layer for future AI integration.

## Added

* Added `app/llm` package.
* Added `LLMClient` protocol and unified LLM response shape.
* Added `MockLLMClient` for local development and tests.
* Added `OpenAILLMClient` with graceful handling for missing API keys or SDK availability.
* Added `LLM_PROVIDER`, `OPENAI_MODEL`, and `OPENAI_API_KEY` configuration.
* Added `PromptContextBuilder` for case, material, fact, analysis, skill, package, and runtime metadata context.
* Added `GET /llm/status`.
* Added `POST /llm/test`.

## Notes

v2.7-A does not replace existing rule-based Fact, Legal Analysis, Report, or Skill Training flows.

The default local provider is `mock`.

## v2.7-B Fact Runtime LLM Integration

v2.7-B connects Fact Runtime to the unified LLM Adapter.

## Added

* Fact Extraction now builds an LLM prompt from case metadata, material content, and optional applied Skill runtime context.
* Skill-aware fact prompts use the Experience Package `fact` prompt when an applied published skill is available.
* Non-skill cases use a default fact extraction prompt.
* Fact Extraction now calls `llm_service.generate_text`.
* Mock LLM responses are parsed into saved fact rows.
* `POST /cases/{case_id}/facts/extract` now returns `llm_provider`, `llm_status`, `skill_used`, and `package_used`.

## Error Handling

If the LLM adapter returns a non-success status, Fact Extraction returns `llm generation failed`.

Missing or empty material files continue to create skipped fact records.

## v2.7-C Legal Analysis LLM Integration

v2.7-C connects Legal Analysis Runtime to the unified LLM Adapter.

## Added

* Legal Analysis now builds an LLM prompt from case metadata, saved facts, and optional applied Skill runtime context.
* Skill-aware analysis prompts use the Experience Package `analysis` prompt when an applied published skill is available.
* Non-skill cases use a default legal analysis prompt.
* Legal Analysis now calls `llm_service.generate_text`.
* Mock LLM responses are parsed into issues, rules, reasoning, conclusion, risk level, and confidence.
* `POST /cases/{case_id}/analysis/run` now returns `llm_provider`, `llm_status`, `skill_used`, and `package_used`.
* Skill-aware analysis rules include a non-empty `Experience Package` rule entry.

## Error Handling

If a case has no facts, Legal Analysis still returns a 400 response asking the caller to run Fact Runtime first.

If the LLM adapter returns a non-success status, Legal Analysis returns `llm generation failed`.

## v2.7-D Report Runtime LLM Integration

v2.7-D connects Report Runtime to the unified LLM Adapter.

## Added

* Report Generation now builds an LLM prompt from case metadata, saved facts, latest legal analysis, and optional applied Skill runtime context.
* Skill-aware report prompts use the Experience Package `report` prompt and report template when an applied published skill is available.
* Non-skill cases use a default report generation prompt.
* Report Generation now calls `llm_service.generate_text`.
* Mock LLM responses produce complete report Markdown with the required five sections.
* Report Generation falls back to the existing template when LLM output does not include all required report sections.
* `POST /cases/{case_id}/reports/generate` now returns `llm_provider`, `llm_status`, `skill_used`, and `package_used`.
* `reports.source_refs` now records `llm_provider`, `llm_status`, `skill_id`, and `package_id` when available.

## Error Handling

If a case has no facts, Report Generation still returns a 400 response asking the caller to run Fact Runtime first.

If a case has no legal analysis, Report Generation still returns a 400 response asking the caller to run Legal Analysis Runtime first.

If the LLM adapter returns a non-success status, Report Generation returns `llm generation failed`.
